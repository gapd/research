開発の目的

ここで開発するModel-View-Controllerは、ウィンドウアプリケーションを構築する際の定石となっているデザインパターンのObserverパターンである。この開発の目的は、デザインパターンの一つを零から構築し、後々に多用できるように抽象化し、ライブラリ化を進めることによって、かかる開発要員が品質の高いソフトウェアを念頭に置き、デザインとプログラミングのスキルを向上させ、ソフトウェア工学的なセンスを磨くことにある。

基本方針

開発に関わるドキュメントおよびプログラムのすべてをHTML文書にしてWebブラウザからアクセス可能にすること。また、アプリケーションのインストールや取り扱いを説明したマニュアルもHTML文書として作成すること。すなわち、この文書からたどれる全文書（このページの最上部にリンクされた全ページ）を完成させなければならない。以下にそれぞれの基本方針を記載する。

  • 設計方針

    アプリケーションの起動時に数枚のウィンドウを開き、後述する各々の機能をそのウィンドウの中に実現すること。オブジェクト指向設計を行い、UML（Unified Modeling Language）を用いて基本設計書（外部設計書）および詳細設計書（内部設計書）にまとめること。

  • 管理方針

    開発に伴う仕様書・設計書・報告書などの全ドキュメントおよび全ソースプログラムをリポジトリ（SCM (Software Configuration Management)：SubversionやGitなどのソフトウェア構成管理）の中に格納し、リビジョン管理（バージョン管理）を行い、開発チームで共有すること。そして、このリポジトリ自体を成果物（納品物）に含めなければならない。

  • 実装方針

    コーディング規約を遵守すること。クラスの名称にはアッパーキャメルケース、メソッドの名称にはロワーキャメルケース、リテラルの名称にはアッパーケースやアッパーキャメルケースを用いること。また、変数やメソッドなどの名称はロワーケースやロワーキャメルケースを基本とする。すべての名称は省略せずに意味を明確に示すものを採用すること。

  • テスト方針

    ユーザの操作に関わるテストケースは設計段階でテスト仕様書にまとめること。xUnit（JUnitやSUnitなど）によるテストケースを行うこと。テスト結果は報告書にまとめ、各々のリリースや大きな修正の直後に全件実施（回帰テスト）を行わなければならない。

  • リリース方針

    オープンソースであること。リポジトリからチェックアウトすることにより、全ソースプログラムを獲得できること。また、ソースプログラムから実行モジュールであるアプリケーションを構築するための手段（MakefileやAntのようなコンパイル・リンク・インストール・テスト・アーカイブ作成などを自動化する手立て）を提供すること。そして、アプリケーションは簡単なコマンドによって起動されなければならない。取り扱い説明書（マニュアル）を必ずリリースに含めなければならない。

基本要件

以下に開発計画書および基本設計書を作成するにあって必要なことを記載する。

  • アプリケーションの用途

    すべてのウィンドウを使用したアプリケーションやグラフィックスを駆使するアプリケーションのベースとなる。ここで構築するモデル（Model）やビュー（View）やコントローラ（Controller）のサブクラスを作成することによって、容易にObserverパターンを利用したアプリケーションを素早く的確に開発できるようになる。

  • 対象となるユーザ

    Observerパターン（Model-View-Controller）は、ウィンドウを用いたアプリケーションを構築するときに、先人が試行錯誤の末にたどりついた最善のデザイン（設計）のパターン（型紙）であり、ウィンドウを使ったソフトウェアを開発しようとする初心者に福音（良い頼り）をもたらす。これからグラフィックスを手がけて、ウィンドウアプリケーションを構築してゆこうとする者たちの一助となるにちがいない。

  • ハードウェア構成

    Name: Apple Macintosh シリーズ　(MacBook / MacBook Pro / Mac mini / iMac / Mac Pro など)
    Processor: Intel Core 2 Duo 2GHz クラス以上（64bit アーキテクチャの Xenon などを含む）
    Memory: 1GB 以上
    Hard Disk: 250MB 以上の空き容量

  • ソフトウェア構成

    OS: Mac OS X Leopard (10.5) / Snow Leopard (10.6) 以上
    開発に用いるプログラミング言語は以下のどちらかとする。
    1) Java: J2SDK (Java 2 Software Development Kit) 1.5.0 以上
    2) Smalltalk: VisualWorks 7.6 (Non-Commercial) 以上

  • 目標と性能

    一つのモデルをビューとコントローラの複数のペアから観測している状態において、モデルの内部状態を変えると、その変更がチェンジ＆アップデート（change & update）の放送機構（broadcast mechanism）により、ビューとコントローラの複数のペアへと伝送される模様を披露（Exampleとして例示）できなければならない。

  • 制限事項

    JavaおよびSmalltalkの標準ライブラリのみを使用し、他の機能拡張ライブラリやツールに組み込まれている機能などを援用してはならない。Macintoshシリーズをターゲットにしているが、実装に用いるJavaもしくはSmalltalkはプラットフォーム非依存性を仮想マシンによって実現しているプログラミング言語である。したがって、開発したアプリケーションはLinuxやWindowsなどでも動作することが望ましい。

開発体制

開発に関わるメンバの一覧を作成し、プロジェクトマネージャ、デザイナ、プログラマ、テスタなどの役割を明確化し、加えて、開発するコンポーネント（部品＝クラス）毎に担当者を決めて、それらを開発計画書の中に明記すること。また、仕様適合検査の責任者を立て、次項の開発スケジュールに示されたマイルストーン（目標達成へ向かう過程の一つの段階）ごとに、設計段階から仕様のチェックをクライアント（開発依頼者＝青木淳）と共に行うこと。開発にあたって不明なことが生じた場合には、必ずクライアントと協議して事を進めなければならない。

開発スケジュール

開発スケジュールを前項の開発体制と共に開発計画書の中に明記し、着手する前にクライアントの認可を受ける（契約を取り交わす）こと。実際の開発はプロジェクト演習の7週間で行うこと。授業（7週間分）の14コマ（1コマは90分）の時間を開発に充当させてよい。開発時間が足りない場合には、適宜、他の空き時間を使って、7週間以内での開発（納品＆リリース）を厳守しなければならない。
